# CLAUDE.md — Agent Instructions for ship-cli

## ⛔ CRITICAL: SECRET HANDLING

**NEVER write real secret values into any file that gets committed to git.**

This includes:
- `.env.example` — MUST use placeholder values like `your-supabase-anon-key`, NEVER real keys
- `.env.local`, `.env` — these are gitignored, but double-check `.gitignore` before committing
- `README.md` — do not paste real API keys, tokens, or JWTs into documentation
- Commit messages — do not include secrets in commit message bodies
- `package-lock.json` — can leak private registry URLs; always use `.npmrc` with public registry
- Any file generated by the `ship` script during scaffolding

Real secrets live in exactly two places:
1. `~/.autodeploy/.env` — GitHub PAT, Cloudflare token (gitignored, local only)
2. `~/.autodeploy/.env.infra` — Dokploy/Supabase keys (gitignored, local only)
3. Dokploy environment variables (set via API, never in code)
4. GitHub Actions secrets (encrypted via API, never in workflow files)

If a secret is accidentally committed:
1. `git filter-branch` to rewrite ALL history
2. `git push --force` to replace remote
3. `rm -rf .git/refs/original/ && git reflog expire --expire=now --all && git gc --prune=now`
4. Verify with a fresh `git clone` that no commit contains the secret
5. **Rotate the leaked credential immediately** — removing from git is not enough

## What This Project Is

`ship` is a single bash CLI (`~/.autodeploy/ship`) that automates the full lifecycle of deploying personal apps:

```
ship new     → scaffold + GitHub + Dokploy + Cloudflare DNS + Tunnel + GitHub Actions
ship deploy  → git push + trigger Dokploy redeploy
ship status  → check if app is live (HTTP probe)
ship destroy → tear down Dokploy + DNS + Tunnel + optionally GitHub repo
```

## Architecture

## Gotchas & Lessons Learned

### 1. GitHub repo creation race condition
GitHub returns 201 for repo creation but the repo isn't immediately pushable. The `push_to_github` function retries up to 5 times with 3-second backoff.

### 2. Cloudflare Tunnel requires TWO steps per subdomain
Just creating a DNS CNAME is not enough. You also need to add an **ingress rule** to the tunnel config pointing the hostname to Dokploy's Traefik (`http://<server-ip>:80`). The `setup_dns` function does both.

### 3. Dokploy domains: https must be false behind Cloudflare
Cloudflare Tunnel terminates TLS at the edge and sends HTTP to origin. If Dokploy's domain is configured with `https: true` + `letsencrypt`, Traefik will redirect HTTP→HTTPS causing an **infinite redirect loop**. Always use `"https": false, "certificateType": "none"`.

### 4. Next.js NEXT_PUBLIC_* vars at build time
Next.js inlines `NEXT_PUBLIC_*` env vars during `next build`. These must be available as **build args** in Dokploy, not just runtime env vars. The `setup_dokploy` function sets both `env` and `buildArgs`.

### 5. Nixpacks vs Dockerfile
Nixpacks auto-detects language and build commands — works out of the box for most stacks. Dockerfiles require manual ARG/ENV handling for build-time vars. Prefer Nixpacks unless you need custom build steps. Add `nixpacks.toml` to pin Node version if needed.

### 6. Private npm registries leak into package-lock.json
If `~/.npmrc` points to a private registry (e.g., AWS CodeArtifact), `npm install` will resolve packages from there and `package-lock.json` will contain those URLs. Always add a project-level `.npmrc` with `registry=https://registry.npmjs.org` to override.

### 7. Supabase Google OAuth flow
The OAuth flow is: App → Supabase Auth → Google → Supabase callback → App callback.
- Google Console redirect URI: `https://supabase-api.yourdomain.com/auth/v1/callback`
- App sets `redirectTo: https://yourapp.yourdomain.com/auth/callback` in `signInWithOAuth`
- Supabase `GOTRUE_URI_ALLOW_LIST` or `GOTRUE_SITE_URL` must allow the app domain

### 8. Supabase RLS is mandatory for user data
Always enable Row Level Security on tables with user data. Create policies scoped to `auth.uid() = user_id`. The service_role key bypasses RLS — only use it server-side.

### 9. Secret rotation after accidental exposure
If keys are committed to git, removing them from history is necessary but **not sufficient**. The keys were exposed the moment they were pushed. Always rotate:
- Supabase: rotate `JWT_SECRET` in docker-compose `.env` → regenerates anon/service keys
- GitHub PAT: revoke and create new token
- Cloudflare: revoke token and create new one
- Dokploy: regenerate API key in Settings → Profile

## Dokploy API Patterns

All Dokploy APIs are tRPC-based:
- **Mutations**: `POST /api/{router}.{procedure}` with JSON body
- **Queries**: `GET /api/{router}.{procedure}?param=value`
- Auth: `x-api-key` header

Key endpoints used by `ship`:
```
POST project.create     → {name, description}
POST application.create → {name, environmentId}
POST application.update → {applicationId, sourceType:"git", customGitUrl, ...}
POST application.saveBuildType → {applicationId, buildType:"nixpacks", ...}
POST application.saveEnvironment → {applicationId, env, buildArgs, buildSecrets, createEnvFile}
POST application.deploy → {applicationId}
POST application.redeploy → {applicationId}
POST application.delete → {applicationId}
POST project.remove → {projectId}
POST domain.create → {applicationId, host, https:false, port, certificateType:"none"}
POST domain.update → {domainId, host, https, port, certificateType}
GET  project.all → lists all projects with environments, apps, status
GET  deployment.all?applicationId=X → deployment history with status and logPath
```

## Cloudflare API Patterns

```
GET  /zones/{zoneId}/dns_records → list DNS records
POST /zones/{zoneId}/dns_records → create CNAME
DELETE /zones/{zoneId}/dns_records/{id} → remove record
GET  /accounts/{accountId}/cfd_tunnel/{tunnelId}/configurations → get tunnel ingress
PUT  /accounts/{accountId}/cfd_tunnel/{tunnelId}/configurations → update tunnel ingress
```

Tunnel ingress is an ordered array; new rules go before the catch-all (last entry). Always preserve the catch-all `http_status:404` rule.

## GitHub API Patterns

```
POST /user/repos → create repo
POST /repos/{owner}/{repo}/hooks → create webhook (unused now, replaced by Actions)
GET  /repos/{owner}/{repo}/actions/secrets/public-key → for encrypting secrets
PUT  /repos/{owner}/{repo}/actions/secrets/{name} → set encrypted secret
DELETE /repos/{owner}/{repo} → delete repo (needs delete_repo scope)
```

Secret encryption uses `pynacl` (libsodium sealed box) with the repo's public key.

## When Modifying the ship Script

1. **Never hardcode secrets** — all values come from `.env` / `.env.infra` at runtime
2. **`.env.example` blocks use single-quoted heredoc** (`<< 'ENVEOF'`) to prevent variable expansion
3. **Test `ship new` end-to-end** after changes — the full flow has many integration points
4. **Push to ship-cli repo** after changes: `cp ~/.autodeploy/ship /path/to/ship-cli-repo/ship`
5. **Verify with `grep -r "eyJ\|ghp_\|sk-\|AKIA" .`** — no real tokens in any committed file
